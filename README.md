# Campsite Web — キャンプ場予約 Web アプリ

## 概要

キャンプサイトの検索・比較から予約、レビュー、お気に入り共有までを一つの Web アプリで扱う**キャンプ場予約サービス**です。利用者向けの公開画面に加え、施設オーナー向けのホストポータルと、承認・問い合わせ対応を行う管理画面を分離し、**MVC（ルート → コントローラ → モデル／ビュー）**で責務を整理しています。認証は Laravel Breeze、データは主に SQLite（本番は永続ディスク上の SQLite など）を想定しています。

| 項目 | 内容 |
|------|------|

**注意:** 無料枠の PaaS ではインスタンスがスリープするため、**初回アクセスやしばらく空けた後の初回は応答が遅くなる**ことがあります。数秒〜数十秒待ってから再読み込みしてください。

## 使用技術

| 区分 | 技術 |
|------|------|
| **フロント** | Blade、Vite 8、Tailwind CSS 3／4 系プラグイン、Alpine.js、Axios（Laravel 標準スタック） |
| **バックエンド** | PHP 8.3、Laravel 13、Eloquent ORM、ミドルウェア（認証・`host`／`admin` ロール、レート制限）、フォームリクエスト／バリデーション、スケジュール／コンソールコマンド |
| **インフラ・運用** | Docker（`Dockerfile` + `docker/entrypoint.sh`）、Render（`render.yaml` Blueprint）、SQLite（開発・本番いずれも利用可能な構成）、Composer／npm ビルドパイプライン |

## 主な機能

- **キャンプサイト一覧・詳細** — タイプ・定員・設備（アメニティ）などで情報表示
- **検索・比較** — 一覧検索、複数サイトの比較画面、カレンダー用の**予約済み日付 API**（重複予約防止の UI 支援）
- **予約** — ログイン＋メール認証済みユーザーの予約作成・一覧・詳細・キャンセル、完了画面
- **レビュー** — 予約に紐づくレビューの投稿・削除
- **お気に入り** — 登録のトグル、共有用トークン URL
- **キャンプ場 Q&A** — 利用者からの質問、ホスト／管理者による回答
- **お問い合わせ** — フォーム送信（レート制限付き）、完了画面
- **周辺情報・補助 API** — 天気・周辺スポット（Overpass プロキシ）、自然言語検索のパース API、チャット API（いずれもルート単位でスロットル）
- **ホストポータル** — 自分のサイト管理、料金・プラン・ブロックアウト、予約の承認
- **管理画面** — キャンプ場の承認、予約ステータス更新、お問い合わせ対応
- **認証・アカウント** — Breeze による登録／ログイン／プロフィール、メール検証

## 工夫した点

- **ロールとルートの分離** — `web.php` で利用者・ホスト（`host`）・管理者（`admin`）のプレフィックスとミドルウェアを分け、コントローラを `App\Http\Controllers\Host` / `Admin` 配下にまとめ、画面と権限の対応を追いやすくした。
- **API とプロキシ** — 外部キーをブラウザに直接渡さないよう、天気・Overpass 等をサーバー側プロキシ経由にし、**レート制限**（`throttle:contact` 等）でコストと悪用リスクを抑えた。
- **予約ドメインの一貫性** — 予約まわりをモデル・Feature テスト（例: `ReservationTest`、`ExpirePendingReservationsTest`）で押さえ、ビジネスルールの変更に強い構造にした。
- **本番アセットの安定化** — リバースプロキシ配下の HTTPS と Vite の整合のため、`TrustProxies` や Vite の**ルート相対アセット**、`public/hot` の扱いを Dockerfile／エントリポイントと合わせて整理（詳細は `docs/DEPLOY_REMEDIATION.md`）。
- **UI** — トップの検索導線、お問い合わせフォーム、共通レイアウトとフッターで、サービスとして一通りの導線を揃えた。

## 苦労した点

- **PaaS 上での Laravel + Vite** — 「ビルド成果物は `public/build` に出るのに、本番だけ CSS が当たらない」問題は、`public/hot` の残存・プロキシヘッダ・asset URL の絶対／相対の組み合わせが原因になりやすく、**症状と原因の切り分けに時間がかかった**。対策をコードとドキュメントに落として再発を防いだ。

## 今後の改善点

- **ホスト／管理者のシード** — デモ用に `is_host` / `is_admin` を立てたユーザーを Seeder で用意し、採用担当がホスト・管理画面まで触れるようにする。
- **E2E テスト** — Playwright 等で予約フロー全体を自動化し、リグレッション検知を強化する。
- **DB の選択** — トラフィック増に備え、本番を MySQL／PostgreSQL に移行する場合のマイグレーションと接続設定の整理。
- **キュー／スケジュールの本番運用** — メール通知・リマインダー等をキューワーカーと cron 連携で安定稼働させる設計の明文化。

## テストアカウント（ローカル／シード利用時）

`php artisan migrate --seed` 実行後、`DatabaseSeeder` により次のユーザーが作成されます（パスワードは Laravel ユーザーファクトリの既定）。

| 種別 | メールアドレス | パスワード | 備考 |
|------|----------------|------------|------|
| 一般ユーザー（例） | `test@example.com` | `password` | メール認証済み状態で作成 |

**ホストポータル**（`/host`）や**管理画面**（`/admin`）は、ユーザーの `is_host` / `is_admin` が true の必要があります。シーダーに含めていない場合は、マイグレーション後に該当ユーザーを DB 上で更新するか、独自シードを追加してください。

## ローカル開発（参考）

```bash
cp .env.example .env
php artisan key:generate
touch database/database.sqlite   # SQLite を使う場合
php artisan migrate
composer run dev   # サーバ・キュー・ログ・Vite を concurrently で起動
```

Docker 利用時はリポジトリの `Dockerfile` および `docker/entrypoint.sh` を参照してください。

```bash
composer test
```

で PHPUnit によるテストを実行できます。
